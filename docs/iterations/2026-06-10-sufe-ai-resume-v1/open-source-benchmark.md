# AI 简历项目开源参考研究

- 更新日期：2026-06-10
- 研究目标：为“面向财经类高校本科毕业生，以上海财经大学为样板学校”的 AI 简历项目筛选可借鉴的开源项目
- 结论先行：这个方向不适合从零发明一整套产品，最稳妥的方式是“成熟简历编辑器 + 中文本地化模板经验 + JD 匹配逻辑 + 简历解析能力 + 标准化数据结构”组合式复用

## 筛选标准

- 与简历编辑、简历解析、JD 匹配、职位数据处理强相关
- 有明确的产品结构或工程实现，不只是概念演示
- 对中文求职场景或财经类校招场景有迁移价值
- 能给当前仓库下一阶段迭代提供直接帮助

## 重点结论

1. 简历编辑器和模板体系，优先参考 `reactive-resume` 与 `open-resume`。
2. 中文求职场景和模板运营经验，优先参考 `resume-design`。
3. JD 匹配逻辑可以吸收 `Resume-Matcher`，但不能直接照搬其海外 ATS 假设。
4. 上传解析阶段可以先借助 `pyresparser` 理解字段抽取思路，但中文 PDF/OCR 仍需单独建设。
5. 数据结构建议向 `resume-schema` 靠拢，避免后续导出、多模板、多版本管理变得混乱。
6. 国内岗位数据和标签体系可以参考 `npe_get_jobs` 的聚合思路，但第一版不建议做自动投递。

## 对当前项目最有用的借鉴组合

### 组合 A：先把核心产品跑通

- `Reactive Resume`：学习编辑器和版本管理
- `Resume Matcher`：学习 JD 匹配反馈闭环
- `JSON Resume Schema`：学习数据结构抽象

适用原因：最适合当前仓库从“文本工具”升级到“产品形态清晰的简历助手”。

### 组合 B：准备进入中文本地化和模板阶段

- `OpenResume`：学习解析与编辑一体化
- `猫步简历`：学习中文模板、导出体验和设计表达

适用原因：能更快补齐“财经类学生愿意持续使用”的体验层，而不是只有一个算法工具壳。

### 组合 C：准备进入上传解析和 JD 数据阶段

- `pyresparser`：学习字段抽取最小链路
- `GetJobs`：学习职位聚合和标签归一化思路

适用原因：能为“上传旧简历 + 目标岗位对比 + 多版本生成”提供底座。

## 针对财经类高校场景的补充判断

### 为什么不能只做一个通用 AI 简历工具

- 财经类高校学生的经历高度同质化，很多人都集中在银行、券商、基金、四大、财务管培和数据分析岗位。
- 他们的问题通常不是“不会写简历”，而是“不会把课程、比赛、社团、实习翻译成岗位语言”。
- 这类用户对“行业词汇准确度”和“岗位适配感”非常敏感，泛化 AI 工具容易生成空泛表述。

### 这个项目真正需要补的差异化能力

- 财经岗位赛道模板：银行管培、券商研究、基金运营、四大审计、财务分析、数据分析
- 中文校招语义包：奖学金、竞赛、学生工作、实训、课程项目、证书
- 财经术语映射：财务分析、估值建模、行业研究、风控、数据处理、Python/SQL/Excel/Power BI 等
- 国内 JD 标签体系：优先围绕 Boss、国聘、实习僧、各大银行校招和券商基金招聘公告构建

## 推荐给当前仓库的落地顺序

1. 先补产品结构，不先补复杂算法。
2. 先做“结构化录入 + 赛道选择 + AI 改写 + JD 对比”的闭环。
3. 再做上传解析，避免一开始就陷入 OCR 和格式兼容泥潭。
4. 暂时不做自动投递，把重点放在“生成更可信、更贴岗的简历版本”。

## 可直接转化为待办的事项

- 建立财经类岗位词典与能力标签体系
- 把仓库内已有结构化表单组件重新接回首页流程
- 增加“目标岗位赛道”选择器
- 增加“同一份经历生成多个岗位版本”的能力
- 为后续上传解析预留统一 resume schema
